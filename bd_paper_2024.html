Q1(II)What is Avro?1M
✓ Answer

Apache Avro is a row-based data serialisation framework developed within the Hadoop project. It uses JSON for defining data types/schemas and serialises data in a compact binary format.

Key Features: Rich data structures, schema evolution (add/remove fields without breaking compatibility), self-describing files (schema embedded in file), language-independent (Java, Python, C++), and compact binary encoding. Widely used in Kafka for message serialisation.

Q1(III)________ is a columnar storage format for Hadoop.1M
✓ Answer

Apache Parquet. It is a columnar storage format optimised for analytics workloads in the Hadoop ecosystem. Data is stored column-by-column rather than row-by-row, enabling efficient compression and encoding. Ideal for queries that access only specific columns (reduces I/O significantly). Supported by Hive, Spark, Pig, and Impala.

Q1(IV)________ is a Cassandra feature that optimizes the cluster consistency process.1M
✓ Answer

Anti-Entropy Repair (or Read Repair). Cassandra uses several mechanisms to maintain consistency:

  • Read Repair: During reads, detects inconsistent replicas and updates stale ones in the background.
  • Hinted Handoff: When a node is down, hints are stored on other nodes and replayed when the node comes back.
  • Anti-Entropy Repair (nodetool repair): Uses Merkle trees to compare data across replicas and synchronise differences.

Q1(VI)The need for data replication can arise in various scenarios like ________.1M
✓ Answer

The need for data replication arises in scenarios like:

  • Fault tolerance: Protecting against hardware failures — if one node fails, data is available on other replicas.
  • Disaster recovery: Maintaining copies across data centres for recovery from site-wide failures.
  • Load balancing: Distributing read requests across multiple replicas to reduce latency.
  • Geographic distribution: Placing data closer to users in different regions for faster access.

Q2Write an explanatory note on MapReduce Work Flow.5 Marks
✓ Answer

The MapReduce workflow is a step-by-step process for distributed data processing in Hadoop:

Input → Splitting → Mapping → Shuffling → Sorting → Reducing → Output

     Input File (HDFS)
          │
    ┌─────┼─────┐
    │     │     │
  Split1 Split2 Split3          ← Input Splitting
    │     │     │
  Map1  Map2  Map3              ← Map Phase (parallel)
    │     │     │
    └─────┼─────┘
          │
    Shuffle & Sort               ← Framework handles this
          │
    ┌─────┼─────┐
  Reduce1 Reduce2               ← Reduce Phase (parallel)
    │     │
  Output1 Output2               ← Final Output (HDFS)
  1. Input Splitting: Input file is divided into fixed-size splits (typically = HDFS block size, 128 MB). Each split is assigned to one Map task.
  2. Map Phase: Each Map task reads its input split, processes records using the user-defined map() function, and emits intermediate key-value pairs.
  3. Combiner (Optional): A local reducer that runs on the Map node to reduce data volume before shuffling — reduces network traffic.
  4. Shuffle & Sort: Framework partitions intermediate output by key, transfers data to Reducer nodes (shuffle), and sorts all values by key (sort).
  5. Reduce Phase: Each Reducer receives all values for its assigned keys, aggregates them using the user-defined reduce() function, and writes final output.
  6. Output: Final results are written to HDFS.

Q4Discuss the advantages of YARN over MapReduce.5 Marks
✓ Answer

YARN (Hadoop 2.x) was introduced to overcome the limitations of MapReduce v1 (MRv1, Hadoop 1.x):

FeatureMRv1 (Hadoop 1.x)YARN (Hadoop 2.x)
Resource ManagementJobTracker handles both resource management AND job scheduling — single point of failureSeparated: ResourceManager (resources) + ApplicationMaster (per-job scheduling)
ScalabilityLimited to ~4,000 nodes (JobTracker bottleneck)Scales to 10,000+ nodes
Multi-TenancyOnly MapReduce can runMultiple frameworks: MapReduce, Spark, Tez, Flink on same cluster
Resource AllocationFixed map/reduce slots — wastefulDynamic containers with configurable CPU/memory
Fault ToleranceJobTracker failure = all jobs failOnly the ApplicationMaster for that specific job needs restart
UtilisationPoor — idle map slots can't be used by reduce tasksBetter — resources are shared dynamically across all applications

Key Advantage: YARN transforms Hadoop from a batch-only MapReduce system into a general-purpose cluster operating system that supports interactive queries (Tez), stream processing (Flink), and in-memory computing (Spark) alongside batch MapReduce.

Q5List some common types of financial frauds prevalent in the current business scenario.5 Marks
✓ Answer

Common Types of Financial Frauds

  • Identity Theft: Stealing personal information (PAN, Aadhaar, credit card numbers) to open accounts, make purchases, or file fraudulent tax returns.
  • Credit Card Fraud: Unauthorised use of credit/debit card information — card skimming, online card-not-present fraud, counterfeit cards.
  • Insurance Fraud: Staging accidents, inflating claims, claiming for pre-existing conditions, or faking death for insurance payouts.
  • Money Laundering: Converting illegally obtained money into legitimate assets through a series of complex transactions (placement → layering → integration).
  • Ponzi/Pyramid Schemes: Investment fraud where returns to existing investors are paid from new investors' money rather than actual profits. Example: Saradha scam in India.
  • Cyber Fraud: Phishing emails, fake websites, vishing (phone scams), smishing (SMS scams) to steal banking credentials.
  • Insider Trading: Trading stocks based on non-public, material information about a company — illegal and unfair to other investors.
  • Invoice/Billing Fraud: Creating fake invoices, inflating expenses, duplicate billing, ghost employee schemes.
  • Tax Fraud: Underreporting income, overclaiming deductions, hiding money in offshore accounts.

Role of Big Data: Big Data analytics can detect fraud patterns by analysing transaction volumes in real-time, identifying anomalies using ML models, and cross-referencing data across multiple sources.

Q6Explain the key features of NoSQL database.5 Marks
✓ Answer
  • Flexible Schema: No rigid table structure required. Documents/records can have different fields. Schema evolves without downtime or migrations.
  • Horizontal Scalability: Scale out by adding more commodity nodes to the cluster. Data is automatically distributed (sharding/partitioning).
  • High Availability: Built-in data replication across multiple nodes/data centres. Automatic failover when nodes go down.
  • High Performance: Optimised for specific access patterns — key-value lookups (sub-millisecond), document queries, graph traversals.
  • Distributed Architecture: Data is distributed across multiple nodes. No single point of failure (masterless in Cassandra).
  • Multiple Data Models: Key-Value (Redis), Document (MongoDB), Column-Family (Cassandra), Graph (Neo4j) — choose the model that fits your use case.
  • Eventual Consistency: Many NoSQL databases favour availability over immediate consistency (AP in CAP theorem). Tunable consistency levels available.
  • Big Data Ready: Designed to handle massive volumes of structured, semi-structured, and unstructured data.
  • Developer Friendly: JSON-like data formats, flexible APIs, and easy integration with modern programming languages.

Q8(a) Short note on HBase. (b) Clusters in Cassandra and YAML file. (c) Replication factor in Cassandra. [15]15 Marks
✓ Answer

(a) Apache HBase [~7 marks]

Apache HBase is an open-source, distributed, column-family NoSQL database built on top of HDFS. It is modelled after Google's Bigtable and provides real-time random read/write access to large datasets.

Key Features

  • Column-Family Store: Data is organised into column families. Each row can have different columns within a family.
  • Linear Scalability: Scales horizontally by adding RegionServers. Can handle billions of rows × millions of columns.
  • Strong Consistency: Unlike Cassandra (eventual), HBase provides strong consistency — reads always return the latest write.
  • Automatic Sharding: Tables are automatically split into regions and distributed across RegionServers.
  • Built on HDFS: Leverages HDFS for fault-tolerant storage with automatic replication.
  • Real-Time Queries: Supports random read/write operations (unlike HDFS which is optimised for batch/sequential).

HBase Architecture

Client → ZooKeeper → HMaster → RegionServer
                                    │
                              ┌─────┼─────┐
                            Region  Region  Region
                              │
                        ┌─────┼─────┐
                     MemStore  HFile  WAL (Write-Ahead Log)
                                │
                              HDFS
  • HMaster: Manages RegionServers, handles schema changes, load balancing.
  • RegionServer: Handles read/write requests for regions assigned to it.
  • Region: A subset of a table's rows (horizontal partition).
  • ZooKeeper: Coordination service — tracks which RegionServer serves which region.
  • MemStore: In-memory write buffer. Flushed to HFiles on HDFS when full.
  • WAL: Write-Ahead Log for crash recovery.

(b) Clusters in Cassandra & YAML File [~5 marks]

Cassandra Cluster: A collection of nodes (machines) that work together to store and manage data. Key concepts:

  • Node: A single Cassandra instance running on a machine.
  • Rack: A logical grouping of nodes (typically in the same physical rack for fault tolerance).
  • Data Centre: A collection of racks. Can represent physical data centres or cloud regions.
  • Cluster: The complete set of all data centres, racks, and nodes.
  • Peer-to-Peer: No master node — all nodes are equal. Uses gossip protocol for inter-node communication.
  • Partitioner: Determines which node stores which data using consistent hashing (partition key → token → node).

YAML File (cassandra.yaml): The primary configuration file for Cassandra. Located in conf/ directory.

  • cluster_name: Name of the cluster (all nodes must have the same name)
  • listen_address: IP address the node listens on for inter-node communication
  • rpc_address: Address for client connections (CQL)
  • seed_provider: List of seed nodes for bootstrapping (joining the cluster)
  • endpoint_snitch: Determines node topology (which rack/DC)
  • partitioner: Hash algorithm for data distribution (default: Murmur3Partitioner)
  • data_file_directories: Paths for storing SSTable data files

(c) Replication Factor in Cassandra [~3 marks]

The replication factor (RF) defines how many copies of each piece of data are stored across the cluster. It is set per keyspace (database).

  • RF = 1: Only one copy — no redundancy. If that node fails, data is lost.
  • RF = 3: Three copies on different nodes — recommended for production. Can tolerate 2 node failures.
  • Rule of thumb: RF should be odd (1, 3, 5) for effective quorum calculations.
CREATE KEYSPACE my_keyspace WITH replication = {
  'class': 'SimpleStrategy',
  'replication_factor': 3
};

-- For multi-data centre:
CREATE KEYSPACE my_keyspace WITH replication = {
  'class': 'NetworkTopologyStrategy',
  'DC1': 3,
  'DC2': 2
};

Consistency Level + RF: With RF=3, a QUORUM read/write requires 2 out of 3 nodes to respond — balancing consistency and availability.

Q11(a) Concept of regions in HBase. (b) Hadoop Integration. (c) Cassandra CQL collections. [5+5+5]15 Marks

(a) Concept of Regions in HBase [5 marks]

A Region in HBase is a horizontal partition of a table containing a contiguous range of rows sorted by row key. Regions are the basic unit of scalability and distribution.

  • Automatic Splitting: When a region grows beyond a threshold (default ~10 GB), HBase automatically splits it into two smaller regions at the midpoint.
  • Region Assignment: Each region is served by exactly one RegionServer. HMaster assigns regions to RegionServers for load balancing.
  • Components: Each region contains a MemStore (in-memory write buffer) per column family, and HFiles (on-disk sorted key-value files).
  • Write Path: Client write → WAL (Write-Ahead Log) → MemStore → (when full) Flush to HFile on HDFS.
  • Read Path: Check MemStore → Check Block Cache → Read HFiles → Merge results.
Table: students
  Row Key Range: A-M → Region 1 (on RegionServer 1)
  Row Key Range: N-Z → Region 2 (on RegionServer 2)

When Region 1 grows too large:
  Region 1 splits → Region 1a (A-F) + Region 1b (G-M)

Meta Table: A special HBase table (hbase:meta) that stores the mapping of row key ranges to regions and their hosting RegionServers. ZooKeeper stores the location of the Meta table.

(c) Cassandra CQL Collections [5 marks]

CQL (Cassandra Query Language) Collections allow storing multiple values in a single column. Cassandra supports three collection types:

1. SET — Unordered collection of unique values

CREATE TABLE users (
  id UUID PRIMARY KEY,
  name text,
  emails set<text>
);

INSERT INTO users (id, name, emails) VALUES (uuid(), 'Alice', {'alice@gmail.com', 'alice@work.com'});

-- Add to set:
UPDATE users SET emails = emails + {'alice@new.com'} WHERE id = ...;

-- Remove from set:
UPDATE users SET emails = emails - {'alice@work.com'} WHERE id = ...;

2. LIST — Ordered collection (allows duplicates)

CREATE TABLE posts (
  id UUID PRIMARY KEY,
  title text,
  tags list<text>
);

INSERT INTO posts (id, title, tags) VALUES (uuid(), 'Big Data', ['hadoop', 'hdfs', 'hadoop']);

-- Append to list:
UPDATE posts SET tags = tags + ['spark'] WHERE id = ...;

-- Prepend:
UPDATE posts SET tags = ['intro'] + tags WHERE id = ...;

3. MAP — Key-value pairs

CREATE TABLE students (
  id UUID PRIMARY KEY,
  name text,
  scores map<text, int>
);

INSERT INTO students (id, name, scores) VALUES (uuid(), 'Bob', {'math': 95, 'science': 88});

-- Add/update entry:
UPDATE students SET scores['english'] = 92 WHERE id = ...;

-- Delete entry:
DELETE scores['science'] FROM students WHERE id = ...;
CollectionOrdered?Duplicates?Access Pattern
SETNo (sorted internally)NoAdd/remove unique values
LISTYes (insertion order)YesAppend/prepend, index access
MAPNo (sorted by key)No (unique keys)Key-based access

Note: Collections are best for small amounts of data (hundreds of values, not thousands). For large collections, use a separate table with clustering columns.
