Big Data Analytics. It encompasses techniques like data mining, machine learning, statistical analysis, and natural language processing to extract meaningful patterns, trends, and insights from massive datasets that exceed the capabilities of traditional RDBMS and BI tools.

Q1(IV)What do you mean by indexing in HDFS?1M
✓ Answer

Indexing in HDFS refers to creating an index (lookup structure) to enable faster data retrieval without scanning entire files. Since HDFS is designed for sequential reads of large files, native indexing is limited. Tools like HBase (built on HDFS) provide row-key based indexing, and Apache Lucene/Solr can be used for full-text indexing on HDFS data.

Q1(V)Which is best described as a programming model that is used to construct Hadoop-based applications that can be scaled up and down?1M
✓ Answer

MapReduce. It is a programming model for processing large datasets in parallel across a Hadoop cluster. It consists of two phases — Map (processes input key-value pairs to produce intermediate pairs) and Reduce (aggregates intermediate pairs by key to produce final output). Scales by adding more nodes.

Q1(VI)What is Pig Latin model?1M
✓ Answer

Pig Latin is the high-level data flow language used by Apache Pig. It provides a set of operators for loading, filtering, grouping, joining, sorting, and storing data. Pig Latin scripts are compiled into MapReduce jobs that run on Hadoop.

Key operators: LOAD, FILTER, FOREACH, GROUP, JOIN, ORDER, STORE, DUMP.

Q1(VII)Which is a popular NoSQL database used for Big Data processing?1M
✓ Answer

Apache Cassandra (or HBase, MongoDB). Cassandra is a distributed, wide-column NoSQL database designed for handling large amounts of data across many commodity servers with no single point of failure. It provides high availability and linear scalability. Used by Facebook, Netflix, Apple, and Instagram.

Q1(VIII)What is the term used for the ability of a system to handle increasing amounts of data and traffic without compromising performance?1M
✓ Answer

Scalability. There are two types: Vertical scalability (scaling up — adding more CPU/RAM to a single machine) and Horizontal scalability (scaling out — adding more machines to the cluster). Hadoop and NoSQL databases are designed for horizontal scalability.

Q1(IX)What is Job Tracker in building blocks of Hadoop?1M
✓ Answer

JobTracker is the master daemon in Hadoop 1.x (MRv1) responsible for:

  • Accepting MapReduce job submissions from clients
  • Splitting jobs into Map and Reduce tasks
  • Assigning tasks to TaskTracker nodes (slaves)
  • Monitoring task progress and re-executing failed tasks

In Hadoop 2.x (YARN), JobTracker's responsibilities are split between ResourceManager (resource allocation) and ApplicationMaster (job management).

Q1(X)Which is a method for analyzing data in which the data is split into smaller subsets and processed in parallel across multiple servers or nodes?1M
✓ Answer

Distributed Computing (or Parallel Data Processing / MapReduce paradigm). Data is partitioned into smaller chunks, distributed across multiple nodes, and processed simultaneously. Each node works on its local subset, and results are aggregated. This is the core principle behind Hadoop's MapReduce framework.

Q1(XII)What are the common input formats in Hadoop?1M
✓ Answer

Common input formats in Hadoop MapReduce:

  • TextInputFormat: Default. Each line is a record; key = byte offset, value = line content.
  • KeyValueTextInputFormat: Each line is split into key-value by a separator (default: tab).
  • SequenceFileInputFormat: Binary format for key-value pairs; supports compression. Used for intermediate MapReduce output.
  • NLineInputFormat: Assigns exactly N lines per split (useful for fixed-size processing).

Q3Explain the Pig Latin application flow.5 Marks
✓ Answer

The Pig Latin application flow describes how a Pig script is processed from writing to execution:

Pig Latin Script → Parser → Logical Plan → Optimizer → Physical Plan → MapReduce Plan → Hadoop Execution
  1. Write Pig Latin Script: User writes data flow operations using Pig Latin operators (LOAD, FILTER, GROUP, JOIN, STORE).
  2. Parser: Checks syntax and semantic correctness. Builds an AST (Abstract Syntax Tree).
  3. Logical Plan: Converts the parsed script into a directed acyclic graph (DAG) of logical operators. No execution yet — just a plan.
  4. Logical Optimizer: Applies optimisations — predicate pushdown, projection pruning, filter reordering — to improve performance.
  5. Physical Plan: Translates logical plan into physical operators that can be executed. Determines the order of operations.
  6. MapReduce Plan: Converts physical plan into one or more MapReduce jobs. Groups operators into Map and Reduce phases.
  7. Execution: MapReduce jobs are submitted to Hadoop cluster. YARN allocates resources, and tasks execute in parallel across DataNodes.
-- Example Pig Latin Script:
data = LOAD '/input/students.csv' USING PigStorage(',') AS (name:chararray, marks:int);
passed = FILTER data BY marks >= 40;
grouped = GROUP passed ALL;
count = FOREACH grouped GENERATE COUNT(passed);
DUMP count;
Q4What does a block in HDFS mean? Explain briefly.5 Marks
✓ Answer

A block in HDFS is the smallest unit of data that HDFS can store and manage. When a file is uploaded to HDFS, it is split into fixed-size blocks and distributed across multiple DataNodes.

Key Characteristics

  • Default Size: 128 MB (Hadoop 2.x+), previously 64 MB in Hadoop 1.x. Configurable via dfs.blocksize.
  • Distribution: Blocks are distributed across different DataNodes in the cluster for parallel processing.
  • Replication: Each block is replicated (default factor = 3) across different nodes/racks for fault tolerance.
  • Last Block: The last block of a file may be smaller than the block size. It only occupies the actual data size on disk (no padding).
  • Metadata: NameNode stores block metadata (location, replication status) in memory; actual data is on DataNodes.
File: report.csv (300 MB)
  → Block 1 (128 MB) → DataNode 1, DataNode 3, DataNode 5
  → Block 2 (128 MB) → DataNode 2, DataNode 4, DataNode 1
  → Block 3 (44 MB)  → DataNode 3, DataNode 5, DataNode 2

Why large blocks? Reduces NameNode memory usage (fewer blocks to track), reduces seek time relative to transfer time, and improves throughput for large sequential reads typical in Big Data processing.

Q5Explain the different features of Hadoop.5 Marks
✓ Answer
  • Open Source: Apache-licensed, free to use with a large community and rich ecosystem (Hive, Pig, HBase, Spark).
  • Scalability: Scales from a single server to thousands of commodity machines by simply adding nodes.
  • Fault Tolerance: Data replicated across multiple nodes (default 3). If a node fails, data is automatically recovered from replicas.
  • Data Locality: Computation is moved to where data resides rather than moving data to computation — drastically reduces network overhead.
  • Distributed Storage (HDFS): Files are split into blocks and distributed across the cluster for parallel access.
  • Distributed Processing (MapReduce/YARN): Processing happens in parallel across all nodes holding data.
  • Cost-Effective: Runs on commodity hardware — no need for expensive specialised storage (SAN/NAS).
  • Flexibility: Can handle any data type — structured, semi-structured, unstructured (text, images, logs, JSON).
  • High Throughput: Optimised for batch processing of massive datasets with sequential read/write patterns.
  • Reliability: Checksums for data integrity; automatic re-replication of under-replicated blocks.
Q6What is the need for Data Locality in Hadoop?5 Marks
✓ Answer

Data Locality is the principle of moving computation to data rather than moving data to computation. It is one of the most important optimisation strategies in Hadoop.

Why is Data Locality Needed?

  • Reduces Network Bandwidth: Moving large datasets (terabytes/petabytes) across the network is extremely slow and expensive. By processing data locally, network traffic is minimised.
  • Faster Processing: Reading data from local disk is much faster than reading over the network. Local reads can achieve ~100 MB/s vs ~10 MB/s over network.
  • Efficient Resource Usage: Network bandwidth is a shared, limited resource. Data locality ensures other jobs aren't starved of bandwidth.
  • Scalability: As cluster size grows, data locality ensures performance scales linearly — each node processes its local data.

Levels of Data Locality (in order of preference)

LevelDescriptionPerformance
Data-LocalTask runs on the same node where the data block is storedBest (local disk read)
Rack-LocalTask runs on a different node in the same rackGood (intra-rack network)
Off-RackTask runs on a node in a different rackWorst (inter-rack network)

Hadoop's scheduler (YARN) prioritises data-local task assignment. It waits briefly for a data-local slot before falling back to rack-local or off-rack.

Q7(a) What is MapReduce in Hadoop? (b) When to use MapReduce with Big Data? (c) Core methods of Reducer. [3+5+7]15 Marks
✓ Answer

(a) What is MapReduce in Hadoop? [3 marks]

MapReduce is a programming model and processing engine in Hadoop for distributed computation on large datasets. It breaks processing into two phases:

  • Map Phase: Takes input key-value pairs, processes them, and outputs intermediate key-value pairs. Runs in parallel across all nodes holding input data.
  • Reduce Phase: Takes all intermediate values for each unique key, aggregates them, and produces final output.
Input → Splitting → Mapping → Shuffling & Sorting → Reducing → Output
                    (parallel)                       (parallel)

(b) When to Use MapReduce with Big Data? [5 marks]

  • Batch Processing: When you need to process massive datasets (TBs/PBs) that don't require real-time results — log analysis, ETL pipelines, data warehouse loading.
  • Text Processing: Word count, inverted index creation, log parsing, sentiment analysis on large text corpora.
  • Data Transformation: Converting data formats, cleaning large datasets, denormalisation.
  • Aggregation: Counting, summing, averaging over massive datasets — sales reports, web analytics, clickstream analysis.
  • Sorting: Distributed sorting of large datasets (TeraSort benchmark).
  • Graph Processing: Iterative algorithms like PageRank (though Spark/GraphX is preferred now).

When NOT to use: Real-time processing (use Spark Streaming/Flink), interactive queries (use Hive/Presto), iterative ML algorithms (use Spark MLlib).

(c) Core Methods of Reducer [7 marks]

The Reducer class in Hadoop MapReduce has three core methods that define its lifecycle:

1. setup(Context context)

  • Called once before any reduce calls.
  • Used for one-time initialisation — loading configuration, opening connections, initialising counters.

2. reduce(KEY key, Iterable<VALUE> values, Context context)

  • Called once for each unique key in the sorted intermediate output.
  • Receives the key and an iterator over all values associated with that key.
  • Contains the core aggregation/processing logic.
  • Writes output using context.write(key, value).

3. cleanup(Context context)

  • Called once after all reduce calls complete.
  • Used for cleanup — closing connections, flushing buffers, writing summary statistics.
import org.apache.hadoop.mapreduce.Reducer;
import java.io.IOException;

public class WordCountReducer extends Reducer<Text, IntWritable, Text, IntWritable> {

    private IntWritable totalCount = new IntWritable();

    @Override
    protected void setup(Context context) throws IOException, InterruptedException {
        // One-time initialization
        // e.g., load configuration, open database connection
    }

    @Override
    protected void reduce(Text key, Iterable<IntWritable> values, Context context)
            throws IOException, InterruptedException {
        int sum = 0;
        for (IntWritable val : values) {
            sum += val.get();          // Aggregate all values for this key
        }
        totalCount.set(sum);
        context.write(key, totalCount); // Output: word → total count
    }

    @Override
    protected void cleanup(Context context) throws IOException, InterruptedException {
        // Final cleanup after all reduce() calls
        // e.g., close connections, flush output
    }
}
Reducer Lifecycle:
  setup() → reduce(key1, [v1,v2,v3]) → reduce(key2, [v4,v5]) → ... → cleanup()
  [once]    [per unique key]           [per unique key]              [once]
Q8(a) What is PIG? Explain its Installation Process. (b) Explain two execution types or modes in PIG. [7+8]15 Marks
✓ Answer

(a) What is Apache Pig? [7 marks]

Apache Pig is a high-level platform for creating programs that run on Hadoop. It uses a scripting language called Pig Latin to express data analysis tasks. Pig abstracts the complexity of writing raw MapReduce Java code.

Key Features

  • High-Level Language: Pig Latin is easier to write than MapReduce Java code. 10 lines of Pig Latin ≈ 200 lines of Java MapReduce.
  • Automatic Optimisation: Pig's compiler optimises the execution plan automatically.
  • Extensibility: Users can write User Defined Functions (UDFs) in Java, Python, or JavaScript.
  • Multi-Query Execution: Multiple Pig Latin statements can be batched and optimised together.
  • ETL Focus: Designed for ETL (Extract, Transform, Load) operations on large datasets.

Installation Process

  1. Prerequisites: Java JDK 1.8+, Hadoop installed and configured, JAVA_HOME and HADOOP_HOME set.
  2. Download: Download Apache Pig binary from https://pig.apache.org/releases.html.
  3. Extract: tar -xzf pig-x.x.x.tar.gz
  4. Set Environment Variables: 
    export PIG_HOME=/path/to/pig-x.x.x
export PATH=$PATH:$PIG_HOME/bin
export PIG_CLASSPATH=$HADOOP_HOME/etc/hadoop
  5. Verify Installation: pig -version — should display Pig version.
  6. Launch Grunt Shell: pig (MapReduce mode) or pig -x local (local mode).

(b) Two Execution Modes in PIG [8 marks]

1. Local Mode

  • Runs on a single machine using the local file system — no Hadoop cluster needed.
  • Launched with: pig -x local
  • All processing happens in a single JVM on the local machine.
  • Use Case: Development, debugging, testing with small datasets before deploying to a cluster.
  • Input/output files are read from/written to the local file system.
-- Local Mode Example:
$ pig -x local
grunt> data = LOAD 'local_file.csv' USING PigStorage(',') AS (name:chararray, age:int);
grunt> filtered = FILTER data BY age > 18;
grunt> DUMP filtered;

2. MapReduce Mode (Hadoop Mode)

  • Default mode. Runs on a Hadoop cluster — Pig Latin scripts are compiled into MapReduce jobs.
  • Launched with: pig or pig -x mapreduce
  • Requires a running Hadoop cluster with HDFS and YARN.
  • Input/output files are read from/written to HDFS.
  • Use Case: Production environments, processing large datasets across the cluster.
-- MapReduce Mode Example:
$ pig
grunt> data = LOAD '/user/input/logs.csv' USING PigStorage(',') AS (ip:chararray, page:chararray);
grunt> grouped = GROUP data BY page;
grunt> counts = FOREACH grouped GENERATE group, COUNT(data);
grunt> STORE counts INTO '/user/output/page_counts';

Q9(a) How do you convert unstructured data to structured data? (b) What is Data preprocessing? (c) Steps for data preparation. [5+3+7]15 Marks
✓ Answer

(a) Converting Unstructured Data to Structured Data [5 marks]

Unstructured data (text, images, audio, social media posts) lacks a predefined schema. Converting it to structured format enables analysis with traditional tools.

Techniques

  • Natural Language Processing (NLP): Extract entities (names, dates, locations), sentiment, and topics from text. Convert to structured tables with columns like entity_name, entity_type, sentiment_score.
  • Regular Expressions: Parse log files and extract specific fields (IP address, timestamp, URL, status code) into structured columns.
  • Image Recognition (Computer Vision): Classify images, detect objects, extract text (OCR) — output as structured labels, coordinates, and text fields.
  • Speech-to-Text: Convert audio recordings to text transcripts, then apply NLP for structuring.
  • Schema-on-Read: Apply structure at query time using tools like Hive (define external tables over raw data in HDFS).
  • Data Parsing: Parse semi-structured formats (JSON, XML, CSV) into tabular format using ETL tools.
-- Example: Parsing Apache log file into structured data
Raw log: 192.168.1.1 - - [21/Jun/2026:10:00:00] "GET /page.html HTTP/1.1" 200 1234

Structured:
| IP            | Date              | Method | URL         | Status | Size |
|---------------|-------------------|--------|-------------|--------|------|
| 192.168.1.1   | 2026-06-21 10:00  | GET    | /page.html  | 200    | 1234 |

(b) What is Data Preprocessing? [3 marks]

Data Preprocessing is the process of transforming raw, noisy, and incomplete data into a clean, consistent format suitable for analysis and modelling. It is a critical step in the data analysis pipeline — "garbage in, garbage out."

Why needed: Real-world data is often missing values, inconsistent, contains duplicates, has outliers, and comes in different formats. Preprocessing ensures data quality and reliability of analysis results.

(c) Steps Required for Data Preparation [7 marks]

  1. Data Collection: Gather data from multiple sources — databases, APIs, files, sensors, web scraping.
  2. Data Cleaning:
    • Handle missing values — imputation (mean/median/mode), deletion, or flagging.
    • Remove duplicates — identify and eliminate redundant records.
    • Fix inconsistencies — standardise formats (date formats, units, naming conventions).
    • Correct errors — typos, invalid values, out-of-range data.
  3. Data Integration: Merge data from multiple sources into a unified dataset. Resolve naming conflicts and schema mismatches.
  4. Data Transformation:
    • Normalisation/Standardisation — scale numerical features to a common range.
    • Encoding — convert categorical variables to numerical (one-hot encoding, label encoding).
    • Aggregation — summarise data at different granularity levels.
  5. Data Reduction: Reduce dataset size while preserving information — dimensionality reduction (PCA), sampling, feature selection.
  6. Data Validation: Verify data quality — check distributions, ranges, relationships, and business rules.
  7. Data Splitting: Split data into training, validation, and test sets (for ML tasks).
Raw Data → Collection → Cleaning → Integration → Transformation → Reduction → Validation → Ready for Analysis
Q10(a) What is Hadoop? (b) Importance of Hadoop in Big Data analytics. (c) Core components of Hadoop. [3+6+6]15 Marks
✓ Answer

(a) What is Hadoop? [3 marks]

Apache Hadoop is an open-source framework for distributed storage and processing of large datasets across clusters of commodity computers. Created by Doug Cutting and Mike Cafarella in 2006, inspired by Google's MapReduce and GFS (Google File System) papers.

Hadoop enables reliable, scalable, and cost-effective processing of petabytes of structured, semi-structured, and unstructured data.

(b) Importance of Hadoop in Big Data Analytics [6 marks]

  • Handles Volume: Can store and process petabytes of data — impossible with traditional RDBMS. HDFS distributes data across thousands of nodes.
  • Handles Velocity: Tools like Kafka + Spark Streaming (in Hadoop ecosystem) process real-time data streams alongside batch processing.
  • Handles Variety: Unlike RDBMS (structured only), Hadoop processes structured (tables), semi-structured (JSON, XML), and unstructured data (text, images, logs) equally well.
  • Cost-Effective: Uses commodity hardware — organisations save millions by avoiding expensive SAN/NAS storage and enterprise software licenses.
  • Fault Tolerant: Automatic data replication ensures no data loss even when hardware fails (which is expected in large clusters).
  • Rich Ecosystem: Hive (SQL), Pig (scripting), HBase (NoSQL), Spark (fast processing), Kafka (streaming), Sqoop (RDBMS import) — comprehensive analytics toolkit.
  • Industry Adoption: Used by Facebook (30+ PB), Yahoo (300+ PB), LinkedIn, Twitter, Amazon — proven at massive scale.
  • Schema-on-Read: Store data first, define schema later — flexibility that traditional schema-on-write databases don't offer.

(c) Core Components of Hadoop [6 marks]

1. HDFS (Hadoop Distributed File System) — Distributed storage layer

  • NameNode (Master): Manages file system metadata — directory tree, file-to-block mapping, block locations. Stores FsImage and EditLog.
  • DataNode (Slave): Stores actual data blocks on local disks. Sends heartbeats and block reports to NameNode.
  • Secondary NameNode: Periodically merges FsImage with EditLog (checkpointing) — not a failover for NameNode.

2. YARN (Yet Another Resource Negotiator) — Resource management layer

  • ResourceManager (Master): Global resource allocation — scheduler and applications manager.
  • NodeManager (Slave): Per-node agent managing containers, monitoring resource usage.
  • ApplicationMaster: Per-application process that negotiates resources from ResourceManager and coordinates task execution.

3. MapReduce — Distributed processing framework

  • Programming model with Map (parallel transformation) and Reduce (aggregation) phases.
  • Writes intermediate results to disk — reliable but slower than in-memory (Spark).

4. Hadoop Common — Shared utilities

  • Java libraries and utilities needed by other Hadoop modules — file system abstractions, I/O, serialisation, networking, configuration.

1. HDFS — Hadoop Distributed File System (Storage Layer)

The distributed storage system that stores data reliably across the cluster.

  • NameNode: Master server that manages file system namespace (directory tree, permissions, block mapping). Stores metadata in FsImage and EditLog. Single point of contact for clients. Does NOT store actual data.
  • DataNode: Slave nodes that store actual data blocks on local disks. Send periodic heartbeats (3-second interval) and block reports (hourly) to NameNode. Serve read/write requests from clients.
  • Secondary NameNode: Performs periodic checkpointing — merges FsImage with EditLog to prevent EditLog from growing too large. NOT a hot standby for NameNode.
  • Blocks: Files are split into 128 MB blocks, replicated 3 times across different DataNodes/racks.

2. YARN — Yet Another Resource Negotiator (Resource Layer)

Cluster resource management and job scheduling. Introduced in Hadoop 2.x to decouple resource management from MapReduce.

  • ResourceManager (RM): Global master — has two components: (1) Scheduler (allocates resources to applications) and (2) ApplicationsManager (accepts job submissions, launches first ApplicationMaster container).
  • NodeManager (NM): Per-node agent — manages containers (isolated execution environments), monitors resource usage (CPU, memory), reports to ResourceManager via heartbeats.
  • ApplicationMaster (AM): Per-application process — negotiates resources from RM, works with NMs to execute and monitor tasks. One AM per submitted job.
  • Container: Logical bundle of resources (CPU + memory) allocated on a node. Each Map/Reduce task runs inside a container.

3. MapReduce (Processing Layer)

Distributed processing framework running on YARN.

  • Map Phase: Input splits → Map tasks (parallel) → Intermediate key-value pairs.
  • Shuffle & Sort: Framework groups all values by key and transfers them to Reducers.
  • Reduce Phase: Aggregates values per key → Final output to HDFS.
  • YARN enables other engines (Spark, Tez, Flink) to also use the cluster — MapReduce is just one processing option.

4. Hadoop Common (Utilities Layer)

Foundation layer providing shared utilities used by all Hadoop modules:

  • File system abstractions (local, HDFS, S3)
  • RPC (Remote Procedure Call) framework for daemon communication
  • Serialisation (Writable interface)
  • Configuration system (XML-based property files)
  • Security (Kerberos authentication)
  • Metrics and logging

5. Client Layer

  • CLI: Command-line tools — hdfs dfs, yarn, mapred.
  • Web UI: NameNode UI (port 9870), ResourceManager UI (port 8088).
  • Java API: Programmatic access via Hadoop Java libraries.
  • REST API: WebHDFS for HTTP-based file operations.
