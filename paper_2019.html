Q3Give examples of parameter passing in Applet.5 Marks
✓ Answer

Parameters are passed to an Applet from HTML using <param> tags inside the <applet> tag. The applet reads them using getParameter().

<!-- HTML: Passing parameters to applet -->
<applet code="GreetApplet.class" width="400" height="150">
  <param name="username"  value="Alice">
  <param name="color"     value="blue">
  <param name="fontsize"  value="24">
</applet>
// Java Applet: Reading parameters
import java.applet.*;
import java.awt.*;

public class GreetApplet extends Applet {
    String username;
    Color  textColor;
    int    fontSize;

    public void init() {
        // Read parameters with defaults if null
        username = getParameter("username");
        if (username == null) username = "World";

        String colorParam = getParameter("color");
        if (colorParam != null && colorParam.equals("blue"))
            textColor = Color.BLUE;
        else
            textColor = Color.BLACK;

        String sizeParam = getParameter("fontsize");
        fontSize = (sizeParam != null) ? Integer.parseInt(sizeParam) : 18;
    }

    public void paint(Graphics g) {
        g.setColor(textColor);
        g.setFont(new Font("Arial", Font.BOLD, fontSize));
        g.drawString("Hello, " + username + "!", 30, 80);
    }
}

Output: Displays "Hello, Alice!" in blue bold size-24 font inside the browser applet area.

Q5Why XML is superior to HTML. Types of XML parsers.2+3 = 5
✓ Answer

XML Superior to HTML (2 marks)

  • Extensible: User-defined tags (HTML has fixed predefined tags).
  • Data-focused: XML transports data; HTML displays it.
  • Strict: Well-formedness rules — all tags closed, case-sensitive.
  • Validatable: Supports DTD and XML Schema for structure validation.
  • Platform-independent: Used across all programming languages and platforms.
  • HTML is presentation-centric; XML is data-centric and machine-readable.

Types of XML Parsers (3 marks)

  • DOM (Document Object Model): Loads entire XML into memory as a tree. Random access, supports read/write. Memory-intensive. Best for small-medium docs.
  • SAX (Simple API for XML): Event-driven streaming parser. Low memory, forward-only, read-only. Best for large files.
  • StAX (Streaming API for XML): Pull-based streaming. Application controls parsing speed. More efficient than SAX for selective processing.
  • JDOM / DOM4J: Java-specific convenience wrappers over DOM/SAX with simpler API.
Q6Explain active attack and passive attack.5 Marks
✓ Answer

Passive Attack

The attacker monitors and eavesdrops on communication without altering the data. The goal is to obtain information secretly without the victim knowing.

  • Eavesdropping: Intercepting network packets to read plaintext data.
  • Traffic analysis: Even without reading content, analysing patterns (who communicates with whom, when, how much) reveals information.
  • Packet sniffing: Tools like Wireshark capture all packets on a network segment.
  • Hard to detect — nothing is modified; no trace left.
  • Defence: Encryption (makes sniffed data unreadable).

Active Attack

The attacker actively modifies, interrupts, replays, or fabricates data or communications.

  • Man-in-the-Middle (MITM): Attacker intercepts and possibly alters communication between two parties.
  • Replay Attack: Valid captured data is re-transmitted later to fool a system.
  • IP Spoofing: Sending packets with forged source IP.
  • DoS/DDoS: Flooding to deny service.
  • Masquerade: Attacker pretends to be an authorised user.
  • Easier to detect — modifies data, generates anomalous traffic.
  • Defence: Authentication, integrity checks (MACs/hashes), firewalls, IDS.

Q7a) HTTP + URL components   b) GET vs POST   c) FTP transmission modes2+1+4+4+4 = 15
✓ Answer

a) HTTP in WWW (2 marks)

HTTP (HyperText Transfer Protocol) is the application-layer protocol that defines how clients (browsers) and servers communicate on the World Wide Web. It uses a request-response model: the client sends an HTTP request; the server returns an HTTP response containing the resource (HTML, image, JSON, etc.). HTTP is stateless — each request is independent. Default port: 80. HTTPS (encrypted via TLS) uses port 443.

URL and Components (1+4 marks)

URL = Uniform Resource Locator — the complete address of a resource on the internet.

  https://www.example.com:8080/products/item.html?id=42&sort=asc#reviews
  └─────┘ └──────────────────┘└────┘ └───────────────────┘└──────────┘└───────┘
  Scheme       Host/Domain     Port         Path              Query    Fragment
  • Scheme: Protocol — http, https, ftp, mailto, file.
  • Host: Domain name (resolved to IP by DNS) or IP address.
  • Port: Optional; default 80 (HTTP), 443 (HTTPS).
  • Path: Location of the resource on the server's file system.
  • Query String: ?key=value pairs — parameters sent to server.
  • Fragment: #section — client-side anchor; browser scrolls to element with that ID.

b) GET vs POST (4 marks)

FeatureGETPOST

CachingCached, bookmarkableNot cached
IdempotentYesNo

c) FTP File Transmission Modes (4 marks)

  • Stream Mode (default): Data transferred as a continuous byte stream. No structure. End-of-file indicated by closing the data connection. Simple and efficient for most transfers.
  • Block Mode: Data divided into blocks. Each block has a 3-byte header (1-byte descriptor flags + 2-byte byte count). Supports restart/resume via EOR (end-of-record) and EOF markers embedded in descriptor.
  • Compressed Mode: Uses run-length encoding — replaces repetitive sequences of identical bytes with a count and one byte. Reduces transfer size. Less common today as OS handles compression.

c) What is an Applet + Parameter Example (5 marks)

A Java Applet is a small Java program embedded in an HTML page. It is downloaded from a web server and executed by the browser's JVM. It extends java.applet.Applet and uses the lifecycle methods: init(), start(), paint(), stop(), destroy().

Making a Class Serializable (3 marks)

A class is made serializable by implementing the java.io.Serializable marker interface. This allows objects of that class to be converted to a byte stream (for network transmission or file storage) and reconstructed.

import java.io.Serializable;

public class Employee implements Serializable {
    private static final long serialVersionUID = 1L; // recommended

    String name;
    int    id;
    double salary;

    public Employee(String name, int id, double salary) {
        this.name = name; this.id = id; this.salary = salary;
    }
}
// To serialise:
ObjectOutputStream oos = new ObjectOutputStream(new FileOutputStream("emp.ser"));
oos.writeObject(new Employee("Alice", 101, 50000));

// To deserialise:
ObjectInputStream ois = new ObjectInputStream(new FileInputStream("emp.ser"));
Employee e = (Employee) ois.readObject();

RMI Remote Object Communication (4 marks)

  Client JVM                              Server JVM
  ┌──────────────┐                        ┌──────────────┐
  │  Client App  │                        │  Server App  │
  │      │       │                        │      │       │
  │   Stub       │── network (TCP) ──────►│  Skeleton    │
  │  (proxy)     │◄── result/exception ───│  (dispatcher)│
  └──────────────┘                        └──────────────┘
        ↑                                        ↓
   RMI Registry                          Actual Object
   (naming service)                      (implements Remote)
  1. Server registers remote object with RMI Registry: Naming.rebind("MyService", obj)
  2. Client looks up registry: MyService s = (MyService) Naming.lookup("rmi://host/MyService")
  3. Client gets a Stub — a local proxy object
  4. Method call on stub → marshals (serialises) arguments → sends over network
  5. Skeleton on server deserialises → calls actual method → returns result
  6. Result marshalled back to client

Use of java.rmi.Remote Interface (4 marks)

java.rmi.Remote is a marker interface (no methods). Any class whose objects need to be remotely accessible must implement this interface. The interface also serves as a tag for the RMI runtime to treat objects as remotely accessible. All methods in a Remote interface MUST declare throws RemoteException to handle network failures.

RMI Program (4 marks)

// 1. Remote Interface
import java.rmi.*;
public interface Calculator extends Remote {
    int add(int a, int b) throws RemoteException;
    int multiply(int a, int b) throws RemoteException;
}

// 2. Server Implementation
import java.rmi.*; import java.rmi.server.*;
public class CalculatorImpl extends UnicastRemoteObject implements Calculator {
    public CalculatorImpl() throws RemoteException {}

    public int add(int a, int b) { return a + b; }
    public int multiply(int a, int b) { return a * b; }

    public static void main(String[] args) throws Exception {
        CalculatorImpl calc = new CalculatorImpl();
        Naming.rebind("Calculator", calc);
        System.out.println("Server running...");
    }
}

// 3. Client
import java.rmi.*;
public class CalculatorClient {
    public static void main(String[] args) throws Exception {
        Calculator calc = (Calculator) Naming.lookup("rmi://localhost/Calculator");
        System.out.println("5 + 3 = " + calc.add(5, 3));
        System.out.println("4 × 6 = " + calc.multiply(4, 6));
    }
}
Q11a) RTP and timing issues   b) PageRank algorithm   c) RTSP and live streaming5+5+5 = 15
✓ Answer

a) How RTP Handles Timing Issues (5 marks)

Real-time data (audio/video) has strict timing requirements. RTP handles this via:

  • Timestamps: Each RTP packet carries a timestamp indicating when the first byte of the payload was sampled. The receiver uses timestamps to reconstruct the correct playback timing — even if packets arrive with varying delays (jitter).
  • Sequence Numbers: 16-bit counter incremented per packet. Allows the receiver to detect lost packets and reorder out-of-sequence packets.
  • Jitter Buffer: Receiver buffers incoming packets for a short time to smooth out network jitter before playing them out at a constant rate.
  • SSRC: Synchronisation Source identifier links audio and video streams so they can be synchronised at playback.
  • RTCP SR (Sender Report): Provides NTP wall-clock time alongside RTP timestamp, allowing inter-stream synchronisation (lip-sync between audio and video).

b) PageRank Algorithm (5 marks)

PageRank is an algorithm developed by Larry Page and Sergey Brin (Google) that ranks web pages based on the quantity and quality of inbound links.

Core Idea: A page is important if many other important pages link to it. Links are treated as "votes."

Formula:

  PR(A) = (1 - d) / N  +  d × Σ [ PR(T_i) / C(T_i) ]

  Where:
  d       = damping factor (typically 0.85)
  N       = total number of pages
  T_i     = pages that link to page A
  C(T_i)  = number of outbound links from page T_i
  PR(T_i) = PageRank of page T_i

Implementation:

  1. Initialise all pages with equal PR = 1/N
  2. Iteratively update each page's PR using the formula above
  3. Repeat until convergence (PR values stabilise)
  4. Pages with higher PR appear higher in search results

The damping factor (0.85) models a "random surfer" who sometimes jumps to a random page instead of following a link (15% chance).

c) RTSP and Live Streaming (5 marks)

RTSP (Real Time Streaming Protocol) is an application-layer protocol that acts like a "remote control" for media servers — it controls the delivery of real-time audio/video streams.

Key Commands:

CommandFunction
OPTIONSAsk what methods the server supports
DESCRIBEGet media description (SDP format)
SETUPEstablish transport parameters (RTP port, UDP/TCP)
PLAYStart/resume stream delivery
PAUSEPause stream delivery
TEARDOWNStop stream and free server resources
 
  Client                              RTSP Media Server
     │── OPTIONS rtsp://server/live.sdp ──►│
     │◄── 200 OK (supported methods) ──────│
     │── DESCRIBE rtsp://server/live.sdp ──►│
     │◄── 200 OK + SDP description ────────│
     │── SETUP (RTP/UDP port 5004) ────────►│
     │◄── 200 OK (session established) ────│
     │── PLAY ────────────────────────────►│
     │◄════ RTP Stream (audio/video) ══════│ (via UDP)
     │── PAUSE ───────────────────────────►│
     │── PLAY  ───────────────────────────►│
     │── TEARDOWN ────────────────────────►│
     │◄── 200 OK ──────────────────────────│

RTSP itself does not deliver the media — it only controls it. Actual media is delivered by RTP over UDP. This separation allows fine-grained playback control while keeping media delivery fast and low-latency.
