Q1(II)What is the use of a namespace in Web Development?1M
✓ Answer

A namespace groups identifiers (element names, attributes, functions, variables) under a unique prefix to avoid naming conflicts when combining content from multiple sources.

  • In XML/XHTML: Namespaces prevent element name clashes when merging documents from different vocabularies: <svg xmlns="http://www.w3.org/2000/svg">
  • In JavaScript: Wrapping code in an object/module to avoid polluting the global scope: const MyApp = { utils: {}, models: {} };
  • In CSS: CSS Modules / BEM methodology to scope class names per component.

Q1(IV)Which feature of Perl provides code reusability? Give an example.1M
✓ Answer

Perl provides code reusability through Modules (and Packages). A module is a .pm file containing reusable subroutines and variables that can be loaded into any script.

# File: MathUtils.pm
package MathUtils;
sub square { return $_[0] ** 2; }

Q1(V)How Does a Race Condition Occur?1M
✓ Answer

A race condition occurs when two or more processes/threads access and modify shared data concurrently, and the final outcome depends on the unpredictable timing/order of their execution.

// Example (JavaScript with shared counter):
let counter = 0;
// Thread 1:  reads counter (0), adds 1, writes back (1)
// Thread 2:  reads counter (0), adds 1, writes back (1)
// Expected: 2 — Actual: 1 (race condition — both read 0)

Prevention: Mutexes, semaphores, atomic operations, or JavaScript's single-threaded event loop (which avoids most race conditions).

Q1(VI)What is VoIP?1M
✓ Answer

VoIP (Voice over Internet Protocol) is technology that converts analogue voice signals into digital packets and transmits them over IP networks (internet) instead of traditional PSTN telephone lines.

Process: Voice → ADC (digitise) → Codec (compress) → RTP packets → UDP/IP → internet → reassemble → DAC → sound. Examples: WhatsApp calls, Skype, Zoom, Google Meet. Cheaper than PSTN, especially for international calls.

Q1(X)What is a JavaBean?1M
✓ Answer

A JavaBean is a reusable Java class that follows specific conventions:

  • No-arg constructor: Must have a public default constructor.
  • Serializable: Must implement java.io.Serializable.
  • Private fields: All properties are private.
  • Getters/Setters: Public methods to access/modify properties (following naming: getName(), setName()).
public class Student implements java.io.Serializable {
    private String name;
    private int age;

    public Student() {}  // no-arg constructor

    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
    public int getAge() { return age; }
    public void setAge(int age) { this.age = age; }
}

Used in JSP via <jsp:useBean>, in Spring Framework, and GUI builders.

Q2Discuss about the layers of OSI Model.5 Marks
✓ Answer
  Layer 7 │ APPLICATION   │ HTTP, FTP, SMTP, DNS, DHCP
  Layer 6 │ PRESENTATION  │ SSL/TLS, JPEG, MPEG, encryption, compression
  Layer 5 │ SESSION       │ NetBIOS, RPC — session establishment/maintenance
  Layer 4 │ TRANSPORT     │ TCP, UDP — end-to-end, segmentation, flow control
  Layer 3 │ NETWORK       │ IP, ICMP, ARP — routing, logical addressing
  Layer 2 │ DATA LINK     │ Ethernet, Wi-Fi — MAC, framing, error detection
  Layer 1 │ PHYSICAL      │ Cables, hubs, signals — bit transmission
  • Layer 7 — Application: Provides network services directly to user applications. Where HTTP, FTP, email live.
  • Layer 6 — Presentation: Data formatting, encryption/decryption, compression. Ensures data is in readable format for the application.
  • Layer 5 — Session: Manages sessions (start, maintain, end) between applications. Handles synchronisation checkpoints.
  • Layer 4 — Transport: Reliable end-to-end delivery. TCP = reliable, ordered; UDP = fast, unreliable. Port numbers live here.
  • Layer 3 — Network: Logical addressing (IP), routing packets across multiple networks.
  • Layer 2 — Data Link: MAC addresses, framing, error detection (CRC), and access to physical medium. Switches operate here.
  • Layer 1 — Physical: Raw bit transmission over cables, fiber, or radio waves. Hubs, repeaters.

Mnemonic: "All People Seem To Need Data Processing" (7→1) or "Please Do Not Throw Sausage Pizza Away" (1→7).

Q3Explain Web Crawler with suitable example.5 Marks
✓ Answer

A web crawler (spider/bot) is an automated program that systematically browses the internet, discovering and indexing web pages for search engines.

Algorithm

1. Start with seed URLs: ["https://example.com"]
2. LOOP:
   a. Dequeue URL from frontier
   b. Check if already visited → skip if yes
   c. Download HTML page (HTTP GET)
   d. Parse HTML → extract all <a href="..."> links
   e. Normalise & filter links (same domain? robots.txt?)
   f. Add unvisited links to frontier queue
   g. Extract page text → send to indexer
3. Repeat until queue empty or limit reached

Example (Python-style pseudocode)

visited = set()
queue   = ["https://wikipedia.org"]

while queue:
    url = queue.pop(0)
    if url in visited: continue
    visited.add(url)

    html = http_get(url)
    index_content(url, html)          # store for search

    links = extract_links(html)
    for link in links:
        if link not in visited and allowed_by_robots(link):
            queue.append(link)
  • robots.txt: File at root of website declaring which paths crawlers should not visit.
  • Politeness: Crawlers add delays between requests to avoid overloading servers.
  • Examples: Googlebot, Bingbot, Slurp (Yahoo).
  • Challenges: Duplicate detection, JavaScript-rendered pages, infinite URL spaces, bot detection.

Q6Explain the working principle of VoIP.5 Marks
✓ Answer
  SENDER SIDE                                    RECEIVER SIDE
  ┌──────────┐                                   ┌──────────┐
  │Microphone│                                   │ Speaker  │
  └────┬─────┘                                   └────▲─────┘
       │ Analog voice signal                          │ Analog
       ▼                                              │
  ┌──────────┐                                   ┌────┴─────┐
  │   ADC    │ digitise (8000 samples/sec)        │   DAC    │ convert back
  └────┬─────┘                                   └────▲─────┘
       │ PCM digital audio                            │
       ▼                                              │
  ┌──────────┐                                   ┌────┴─────┐
  │  Codec   │ compress (G.711/G.729/Opus)        │  Codec   │ decompress
  └────┬─────┘                                   └────▲─────┘
       │                                              │
       ▼                                              │
  ┌──────────┐                                   ┌────┴─────┐
  │ Packetise│ RTP header + audio data            │  Jitter  │ reorder
  └────┬─────┘                                   │  Buffer  │
       │                                         └────▲─────┘
       ▼                                              │
  ┌───────────────────────────────────────────────────┴──┐
  │              UDP / IP network (internet)              │
  └──────────────────────────────────────────────────────┘
  1. Capture: Microphone captures voice as analogue signal.
  2. Digitise (ADC): Analogue-to-Digital Converter samples at 8000 Hz (G.711) or higher.
  3. Compress (Codec): G.711 = 64kbps (uncompressed); G.729 = 8kbps (compressed). Codec removes silence, reduces redundancy.
  4. Packetise: Compressed audio chunks wrapped in RTP header (sequence number, timestamp, SSRC).
  5. Transmit: RTP packets sent via UDP/IP over the internet (low latency).
  6. Jitter Buffer: Receiver buffers packets briefly (50–150ms) to smooth variable arrival times.
  7. Decompress → Playback: Codec decompresses; DAC converts digital back to analogue; speaker plays audio.

Signalling (call setup/teardown) handled separately by SIP or H.323, independent of RTP media flow.

Q7Discuss the working principle of RTP, RTCP, RTSP, IPTV. Explain the role of SCO. [12+3]15 Marks
✓ Answer

1. RTP — Real-time Transport Protocol (3 marks)

RTP carries real-time audio/video over UDP. It does not guarantee delivery — it accepts some packet loss for low latency.

RTP Header (12 bytes minimum):
┌────┬────┬────┬────┬──────────┬───────────┬──────────┬──────────┐
│ V  │ P  │ X  │ CC │    M     │    PT     │ Seq Num  │Timestamp │
│2bit│1bit│1bit│4bit│   1bit   │   7bit    │  16bit   │  32bit   │
└────┴────┴────┴────┴──────────┴───────────┴──────────┴──────────┘
SSRC (32 bit) — identifies the synchronisation source
  • Sequence number: Detects packet loss and reorders out-of-order packets.
  • Timestamp: Enables playback synchronisation and jitter calculation.
  • Payload Type (PT): Identifies codec (e.g., G.711 = PT 0).
  • SSRC: Unique stream identifier when multiple streams exist.

2. RTCP — RTP Control Protocol (3 marks)

Companion protocol to RTP. Sends periodic control packets on a separate port (RTP port + 1).

Packet TypeSent ByContains
SR (Sender Report)Active senderPacket count, byte count, timestamp, SSRC
RR (Receiver Report)Passive receiverFraction lost, cumulative lost, jitter, delay
SDESAny participantSource description (CNAME, NAME)
BYELeaving participantStream termination notice

Enables QoS monitoring: if packet loss > 5%, receiver can request lower bitrate or switch codec.

3. RTSP — Real Time Streaming Protocol (3 marks)

Application-layer control protocol — like a TV remote for media servers. Does NOT carry media itself.

Client                          Media Server
  │── OPTIONS rtsp://server/stream ──────────►│  (capabilities)
  │◄── 200 OK ─────────────────────────────────│
  │── DESCRIBE ────────────────────────────────►│  (SDP description)
  │◄── 200 OK (SDP: codec, ports) ─────────────│
  │── SETUP track=audio ───────────────────────►│  (establish session)
  │◄── 200 OK (Session: 12345) ────────────────│
  │── PLAY Session: 12345 ─────────────────────►│  (start streaming)
  │◄── 200 OK ─────────────────────────────────│
  │════════════ RTP media stream ═══════════════│
  │── PAUSE Session: 12345 ────────────────────►│
  │── TEARDOWN Session: 12345 ─────────────────►│  (end session)

Uses SDP (Session Description Protocol) in DESCRIBE response to negotiate codec, IP addresses, and RTP ports.

4. IPTV — Internet Protocol Television (3 marks)

Delivers television content over IP networks instead of traditional cable/satellite broadcast.

IPTV Architecture:
┌──────────┐    IGMP Join     ┌──────────────┐    RTP Multicast    ┌──────────┐
│  Set-Top │◄────────────────►│  IPTV Head   │────────────────────►│  CDN /   │
│   Box    │    RTSP Control  │    End       │    RTSP (VoD)       │  Edge    │
└──────────┘                  └──────────────┘                     └──────────┘

Live TV:  IP Multicast (IGMP) — one stream to many subscribers
VoD:      IP Unicast + RTSP — individual stream per viewer
  • Live TV: Uses IP multicast — efficient bandwidth (one stream, many receivers).
  • Video on Demand (VoD): Uses unicast + RTSP — each viewer gets individual stream.
  • Protocols: IGMP (multicast group join), RTP (media), RTSP (control), MPEG-TS (transport).

5. Role of SCO — Synchronous Connection Oriented Link (3 marks)

SCO is a Bluetooth link type (not IP-based) that provides synchronous, reserved-bandwidth connections for voice.

  • Reserves fixed time slots in the Bluetooth frame — guaranteed bandwidth and latency.
  • Used in Bluetooth headsets and hands-free kits for voice calls.
  • Provides 64 kbps full-duplex voice channel (same as PSTN).
  • Re-transmits lost packets immediately — prioritises voice quality over data throughput.
  • Contrast with ACL (Asynchronous Connection-Less): best-effort data link without reserved slots.

In VoIP over Bluetooth: SCO carries the voice between phone and headset; RTP carries voice over IP between endpoints.

Q8Explain IP masquerading with example network. Role of NAT. Differentiate IPv4 and IPv6. Working process of POP3. [3+3+3+6]15 Marks
✓ Answer

(a) IP Masquerading with Example Network [3 marks]

IP Masquerading is a form of NAT where all devices on a private LAN share a single public IP address. Outgoing packets have their source IP rewritten to the router's public IP; incoming responses are mapped back using port numbers.

                    Internet (Public)
                         │
              Public IP: 203.0.113.50
                         │
                  ┌──────┴──────┐
                  │   Router    │  ← performs IP Masquerading (NAT)
                  │ 192.168.1.1 │
                  └──────┬──────┘
           Private LAN (192.168.1.0/24)
          ┌──────┬───────┼───────┬──────┐
       PC-A    PC-B    Phone   Laptop
    .1.10    .1.20    .1.30   .1.40

Outgoing: PC-A (192.168.1.10:5000) → Router rewrites to (203.0.113.50:60001)
Incoming: (203.0.113.50:60001) → Router maps back to PC-A (192.168.1.10:5000)

(b) Role of NAT [3 marks]

  • IPv4 conservation: Many private devices share one public IP — delays IPv4 exhaustion.
  • Security: Internal network topology hidden from the internet — inbound connections blocked by default.
  • Flexibility: ISP can change public IP without reconfiguring internal network.
  • Types: Static NAT (1:1), Dynamic NAT (pool), PAT/NAPT (many:1 via ports — most common).

(d) Working Process of POP3 Protocol [6 marks]

POP3 (Post Office Protocol v3) retrieves email from a mail server to a local client. Port 110 (995 with SSL/TLS).

Client (MUA)                    Mail Server
  │── TCP connect port 110 ──────────────────►│
  │◄── +OK POP3 server ready ─────────────────│
  │── USER alice@example.com ─────────────────►│
  │◄── +OK User accepted ─────────────────────│
  │── PASS secretpassword ────────────────────►│
  │◄── +OK Logged in ─────────────────────────│
  │── STAT ───────────────────────────────────►│  (count + total size)
  │◄── +OK 3 4521 ────────────────────────────│  (3 messages, 4521 bytes)
  │── LIST ───────────────────────────────────►│  (list all messages)
  │◄── +OK 3 messages ────────────────────────│
  │◄── 1 1200 ────────────────────────────────│
  │◄── 2 2100 ────────────────────────────────│
  │◄── 3 1221 ────────────────────────────────│
  │◄── . ─────────────────────────────────────│
  │── RETR 1 ─────────────────────────────────►│  (retrieve message 1)
  │◄── +OK 1200 octets ───────────────────────│
  │◄── [full email content] ──────────────────│
  │◄── . ─────────────────────────────────────│
  │── DELE 1 ─────────────────────────────────►│  (mark for deletion)
  │◄── +OK Message deleted ───────────────────│
  │── QUIT ───────────────────────────────────►│
  │◄── +OK Bye ───────────────────────────────│  (commits deletions)
  • Authorization state: USER + PASS authentication.
  • Transaction state: STAT, LIST, RETR, DELE commands.
  • Update state: QUIT commits all DELE operations — messages removed from server.
  • Default behaviour: Downloads and deletes from server (offline access). Contrast with IMAP which keeps mail on server for multi-device sync.

Browser loads HTML page

import java.applet.Applet;
import java.awt.*;

public class MyApplet extends Applet {
    private int count = 0;

    public void init() {
        setBackground(Color.WHITE);

    }

    public void start() {
        System.out.println("start(): Applet started");
    }

        g.drawString("Hello MAKAUT! Count: " + count, 50, 50);

    }

    public void stop() {
        System.out.println("stop(): Applet stopped");
    }

    public void destroy() {
        System.out.println("destroy(): Applet destroyed");
    }

(c) Create and Store Cookies — Illustration [4 marks]

Server-side (HTTP response header):

HTTP/1.1 200 OK
Set-Cookie: session_id=abc123; Expires=Wed, 31 Dec 2025 23:59:59 GMT; Path=/; HttpOnly; Secure
Set-Cookie: theme=dark; Max-Age=86400; Path=/

Client-side (JavaScript):

// Create cookie
document.cookie = "username=Alice; expires=Fri, 31 Dec 2025 23:59:59 GMT; path=/";
document.cookie = "visits=5; max-age=3600";  // expires in 1 hour

// Read all cookies
console.log(document.cookie);
// "username=Alice; visits=5"

// Read specific cookie
function getCookie(name) {
    const match = document.cookie.match(new RegExp("(^| )" + name + "=([^;]+)"));
    return match ? match[2] : null;
}
console.log(getCookie("username"));  // "Alice"

// Delete cookie (set expiry in past)
document.cookie = "username=; expires=Thu, 01 Jan 1970 00:00:00 GMT; path=/";

Storage flow: Server sends Set-Cookie → Browser stores in cookie jar → Browser sends Cookie: header with every subsequent request to matching domain/path.

(c) Role of Congestion Control in Network [5 marks]

Congestion occurs when traffic exceeds network capacity — packets queue, drop, and retransmit, worsening the problem.

Normal:  [Sender] ────► [Router] ────► [Receiver]   (low delay)

Congested: [Sender] ═══► [Router ■■■] ──X──► [Receiver]
                        (queue full, packets dropped)

TCP Congestion Control mechanisms:

  • Slow Start: cwnd starts at 1 MSS, doubles each RTT until threshold or loss detected.
  • Congestion Avoidance: After threshold, cwnd increases linearly (+1 MSS per RTT).
  • Fast Retransmit: 3 duplicate ACKs → retransmit lost segment immediately (don't wait for timeout).
  • Fast Recovery: After fast retransmit, halve cwnd and continue (don't go back to slow start).
  • AIMD: Additive Increase, Multiplicative Decrease — fair sharing of bandwidth.

Network-level: RED (Random Early Detection) — routers drop packets probabilistically before queue is full to signal congestion early. ECN (Explicit Congestion Notification) — marks packets instead of dropping.
